§0 Abstract

Content teams cannot manually review every page on a site to decide which ones need attention, so this project asks a narrow question: which pages should be prioritized for review, based on search performance signals? Using an anonymized sample of FlyRank's production content-performance data, we built a transparent rule-based baseline and compared it against a Random Forest Regressor trained on engagement and traffic features. Under an honest, client-grouped validation split, the model reached an R² of 0.698 (MAE 8.66) — a meaningfully lower number than the 0.882 a naive random split reports, confirming that grouped validation is required to avoid overstating performance. The model's strongest signals were sustained impression history and historical clicks, and it produced a ranked, reason-coded action queue meant to support — not replace — a human editor's judgment.

§1 Introduction / Problem Statement

Unit of analysis: one content page, belonging to one client, scored once per analysis run.

Decision supported: which pages a client's content or SEO team should prioritize for review and refresh first — instead of manually reviewing every page on a site.

Output: a ranked priority score, a reason code (e.g. STALE_LOW_CTR), and a recommended action (Refresh Immediately, Monitor, etc).

Action a human takes: an editor reviews the top of the queue and decides whether to refresh, expand, or leave a page alone.

Cost of a wrong call: resource misallocation, not safety risk — a team may refresh pages that didn't need it, while pages genuinely losing visibility keep declining unnoticed.

Why ML helps: a single fixed rule can only combine a couple of signals at once. Page performance is driven by many interacting features — impressions, clicks, position, freshness, length, engagement — and a model can learn how these interact instead of relying on one formula.

§2 Data

Source: the FlyRank ML Internship dataset, drawn from the production search-performance warehouse (fact_content_daily_performance).

3.61Mrows with GSC data available
30,000rows in the modeling sample

The modeling work in this report runs on the provided anonymized export, content_refresh_anonymized.csv — 30,000 rows × 44 columns, one row per anonymized content/client pair, covering the March 2026 window (2026-03-01 to 2026-03-31, 31 days).

Excluded columns, and why

  • trend_direction, trend_pct — label-derived from the same window as the target; including them leaks future information into the prediction.
  • client_id, content_id — used only for grouping and identification, never as model features.
  • Any manually engineered "future" columns — excluded on principle to keep every feature available strictly before the decision point.
Leakage caught in testing: a label-derived feature was deliberately added as a test during development. It pushed R² to a perfect 1.0 — a textbook leakage signature — confirming the need to strip label-derived fields from the feature set. That feature was permanently removed.

Privacy: all client and content identifiers are pseudonymous hashes (e.g. client_73cda7b4e4f265ea). No real client names, URLs, or raw search queries appear anywhere in this dataset or report.

§3 Methodology

Baseline (the transparent rule)

Score a page higher if it has gone a long time without an update, has a low click-through rate, and still receives meaningful impressions — that combination signals wasted visibility.

baseline_score = (days_since_last_update / 30)
               + (100 - ctr * 100)
               + (impressions_90d / 1000)

Every input is available before the decision is made. Two signal checks supported the rule: CTR fell steadily from 1.57 (positions 1–5) to 0.15 (positions 51+) — confirmed. Impressions vs. search volume was mixed: the highest search-volume bucket did not have the highest impressions, meaning other factors (position, CTR, content quality) also drive traffic.

Model

Random Forest Regressor (n_estimators=200, random_state=42), chosen to capture non-linear feature interactions a single rule cannot, while staying interpretable via feature importances.

Target / proxy (one sentence): the model predicts the rule-based baseline_score — a composite proxy for review-need — rather than a directly observed business outcome.

Final features: all remaining numeric columns after removing leakage fields — including clicks_90d, word_count, char_count, avg_position, engagement_rate, scroll_rate, content_age_days, days_with_impressions, impressions_last_30d, impressions_prev_30d.

Validation design

Two splits were compared deliberately: a naive random 80/20 split, and a client-grouped split (GroupShuffleSplit) that keeps every client entirely in either train or test — simulating performance on a client the model has never seen, the realistic deployment case.

Leakage checks

  • No client identifier used as a model feature.
  • No target or future-window information included in the feature set.
  • Features are historical observations available before prediction time.
  • Validation performed with grouped splitting to reduce cross-client leakage.

Error analysis

The model is most reliable for clients with longer historical data. For clients with sparse or very new history, predictions are noisier — expected, since fewer signals exist for the model to generalize from.

Interpretation

Sustained visibility history and click volume dominate the prediction far more than content length or exact ranking position — suggesting that consistency of impressions over time is a stronger review-priority signal than any single snapshot metric.

Negative result worth keeping: search volume alone was not a clean predictor of impressions. The highest search-volume bucket did not correspond to the highest impression bucket — search volume by itself is not a reliable stand-in for actual visibility.

§5 Limitations & Honest Framing

  • This work shows observed correlations, not causation — it does not prove refreshing a page causes better performance.
  • The target is a proxy score (the baseline rule), not a directly measured business outcome like revenue or conversions.
  • Client histories are unbalanced — some clients have only GSC or only GA4 data, and some started later than others.
  • The analysis window is limited to March 2026 and a 30,000-row sample; it does not capture full-year seasonality.
  • The honest R² (0.698, grouped split) should always be trusted over the optimistic random-split number (0.882) — this gap is the clearest limitation in this report.
  • Nothing here predicts search engine algorithm behavior; it only reflects patterns in FlyRank's own historical performance data.

§6 Ranked Recommendations

An editor working this queue tomorrow would move top-down through the following:

PriorityActionReason codeWhen it applies
HighRefresh ImmediatelyLOW_CTR_HIGH_IMPRESSIONSStrong visibility, poor CTR
HighUpdate ContentSTALE_CONTENTOld, hasn't been touched recently
MediumImprove SEOLOW_TRAFFICMetadata / headings / targeting likely weak
MediumExpand ContentTHIN_CONTENTToo short to compete
LowMonitorSTABLE_PERFORMANCEPerforming acceptably, no action yet

Confidence: high for clients with rich history; lower for new or sparse clients. This queue is decision-support only — every recommendation needs human review before action: check current analytics, confirm no conflicting work is planned, and rule out a seasonal or temporary spike.

Never automate: publishing or deleting content, legal or compliance-sensitive edits, medical/financial/policy content, or brand-voice decisions — these always require a human.
